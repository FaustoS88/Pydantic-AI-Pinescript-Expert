# RAGAS Tier 2 Comparison — Contextual Retrieval

## Summary

Tier 2 adds **index-time LLM enrichment** on top of the Tier 1 retrieval pipeline. Each chunk gets a
one-sentence contextual prefix generated by an LLM at crawl time, following
[Anthropic's Contextual Retrieval](https://www.anthropic.com/news/contextual-retrieval) approach (Sep 2024).

**Result: F=0.833 (+7.6% over Tier 1, +6.9% over original baseline)**

Code generation improved +21%. Context Relevance hit 0.92 — retrieval is finding the right chunks.

## By Category

| Category | Baseline | Tier 1 | Tier 2 | T1→T2 delta |
|----------|----------|--------|--------|-------------|
| function_lookup | 0.84 | 0.83 | **0.89** | +7% |
| conceptual | 0.99 | 0.98 | **0.97** | -1% |
| code_generation | 0.51 | 0.53 | **0.64** | +21% |
| complex | 0.78 | 0.75 | **0.84** | +12% |

### Why code_generation improved most (+21%)

The original weakness: the DB contains function reference docs but few complete working examples.
When asked to generate code, the LLM would retrieve parameter signatures but hallucinate the
scaffolding around them.

Contextual prefixes fix this in two ways:
1. Each chunk now embeds with a sentence like *"PineScript strategy.entry() controls long/short position
   opening with id, direction, qty, limit, stop, and when parameters"* — the embedding now carries
   semantic intent, not just raw text
2. The retriever finds chunks about **strategy patterns** even when the query says *"RSI crossover
   strategy"* (not an exact match for any function name)

### Why cross-encoder reranking is still disabled

`ms-marco-MiniLM-L-6-v2` is trained on web search queries. It promotes textual fluency over
code-relevant passages, demoting function signatures and code examples in favor of prose. This caused
a **-3.7% regression** in the Tier 1 rerank run. A code-specific reranker is needed for further gains.

## Context Relevance Now Available

Context Relevance (CR) measures whether the retrieved chunks were actually relevant to the question.
It was unavailable in early Tier 1 runs due to a RAGAS 0.4 API incompatibility, now resolved.

| Category | CR |
|----------|----|
| function_lookup | 0.85 |
| conceptual | 1.00 |
| code_generation | 0.93 |
| complex | 0.90 |
| **Overall** | **0.919** |

CR=0.92 means the retrieval pipeline is finding the right chunks 92% of the time. The remaining
faithfulness gap (F=0.833 vs theoretical max 1.0) is mostly a generation problem — the docs don't
contain complete strategy templates, so the LLM still partly fills gaps from training knowledge.

## Worst Performers (bottom 5 by context precision)

| Question | CP | F | Root Cause |
|----------|----|---|------------|
| `ta.sma() vs ta.ema() difference` | 0.25 | 0.96 | Retrieves each function separately, not the comparison page |
| `ta.ema() parameters` | 0.50 | 1.00 | Good answer despite imperfect precision |
| `Bollinger Band breakout strategy` | 0.50 | 0.18 | No strategy template in docs — LLM hallucinates entry/exit |
| `RSI + MACD + BB combined strategy` | 0.50 | 0.61 | Multi-concept — partial context only |
| `Adaptive moving average (EMA↔SMA)` | 0.50 | 0.73 | Niche topic, limited coverage |

The top-2 failures are **content gaps** (no strategy templates in the docs), not retrieval failures.
This points to the next improvement: synthetic code examples.
